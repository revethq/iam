# Change: Add Permission Discovery Endpoint

## Why

Downstream systems using `permission-web` have no way to advertise the complete set of permissions they understand. A well-known discovery endpoint allows tooling, admin UIs, and other services to enumerate all available actions and resource types without hardcoding knowledge of each service. By aggregating permissions from the downstream system and its transitive dependencies, the endpoint provides a single source of truth for what a running instance can authorize.

## What Changes

- **NEW**: `PermissionDeclaration` domain model representing a single declared permission (action, description, resource type pattern)
- **NEW**: `PermissionManifest` domain model grouping declarations by service name
- **NEW**: `PermissionProvider` CDI interface for modules/services to declare their permissions
- **NEW**: `PermissionRegistry` aggregator that collects all `PermissionProvider` beans from the classpath
- **NEW**: `WellKnownPermissionsResource` JAX-RS endpoint at `/.well-known/revet-permissions` that returns the combined manifest

## Impact

- Affected specs: Creates new `permission-discovery` capability
- Affected code:
  - `permission/` module - `PermissionDeclaration`, `PermissionManifest`, `PermissionProvider`, `PermissionRegistry`
  - `permission-web/` module - `WellKnownPermissionsResource` and response DTOs
- Dependencies: None (uses standard CDI injection; Jandex indexes already generated by root build)

## Design Decisions

1. **CDI-based aggregation**: Each module ships an `@ApplicationScoped` `PermissionProvider` bean. CDI + Jandex auto-discovers all implementations from the classpath, including transitive dependencies. No classpath scanning hacks needed.
2. **Domain models in `permission/` module**: `PermissionDeclaration`, `PermissionManifest`, and `PermissionProvider` live in the core `permission` module so any module in the dependency tree can implement `PermissionProvider` without depending on `permission-web`.
3. **No persistence**: Permissions are declared in code, not stored in the database. The endpoint is a read-only view of what is compiled into the running application.
4. **Grouped by service**: The response organizes permissions by service name (from `PermissionManifest.service`), making it easy for consumers to understand which service contributes which permissions.
5. **Future annotation scanning deferred**: A Quarkus build-step that auto-generates a `PermissionProvider` from `@RequiresPermission` annotations is a natural future enhancement but not needed for the initial implementation.
